Understanding Zoom Content

Intent

The intent is to allow the browser zoom function to zoom content to 400% without requiring scrolling in the direction of the text. For western languages, zooming page content will not require horizontal scrolling. Goal: Authors to responsively support the resizing and reflow of content as allowed by the browser.

Zoom functionality in browsers allows users to increase the size of all content. When appropriately authored, the content can reflow (wrap) to stay within the windows boundaries (viewport). Spatial relationships can vary, but all content and functionality continues to be available.

The size 400% was picked as the largest size a standard website can currently get to using responsive design. If the user needs to increase content more than 400%, then a complete linearisation would be necessary, which completely overrides the layout of the website.

Technologies like HTML/CSS, PDF and ePub have methods of reflowing content to fit the width of the user-agent window (viewport). When zooming in on HTML/CSS content, pixels increase in size, so an element that is 200 pixels wide at 200% takes 400 pixels of the width on screen.

NB: Mobile devices are by necessity more limited in how they can resize content. This success criteria is to ensure the content is capable of resizing to a reasonable degree. In practice people will need to use a device or user-agent capable of reflowing content such as a desktop or laptop computer. In short, users will be able to get a "mobile view" of content on a desktop sized monitor that only scrolls in one direction.

The impact of horizontal scrolling can increase the effort required to read by 40-100 times, so avoiding scrolling should be the aim whenever feasible.

The testing parameters are more defined than WCAG 2.0's 1.4.4, where the test should start with a browser at 1280px wide, and then zoom in to 400%. In desktop browsers that means your view is then 320px wide, equivalent to a mobile device. Many sites currently cater to that range of sizes, and that number will only increase as more sites update to be mobile friendly. What is not achieved by being mobile-friendly is testing under landscape orientation, rather than assuming a (portrait) mobile device.

It is also possible to meet this success criteria in PDF files by using the reflow feature of Acrobat Reader, which works well on many tagged PDFs with a defined order and structure. ePub files are designed to reflow so should be able to meet this criteria easily.

Benefits

  • Responsive design and using native browser functions mitgates the need for screen magnification software and resultant loss of context from horizontal scrolling for users needing 400% or less content size.
  • There are 6 times more people with low vision than are blind. (Wikipedia - As of 2015 there were 940 million people with some degree of vision loss.[4] 246 million had low vision and 39 million were blind.[3])
  • A significant proportion of people with low vision need more than a 200% increase in the size of content. From the Webaim survey 25% of low vision users indicated needing magnification to 400% or more, which is probably more than the total number of screen reader users. (It is 42% for 300% or more).
  • Horizontal scrolling makes reading difficult for people with low vision, or cognitive disorders, or people on mobile divices. Horizontal scrolling can increase the effort required to read by 40-100 times, so avoiding scrolling should be the aim whenever feasible.
  • Some people need to increase the size of all content in order to perceive information.

Techniques

Sufficient

Techniques that are sufficient to meet the Guideline or Success Criterion.

Situation

  • G146: Using liquid layout and ''AND'' using measurements that are relative to other measurements in the content by using one or more of the following techniques:
    • C12: Using percent for font sizes (CSS) OR
    • C13: Using named font sizes (CSS) OR
    • C14: Using em units for font sizes (CSS) OR
    • C24: Using percentage values in CSS for container sizes (CSS) OR
    • FLASH33: Using relative values for Flash object dimensions (Flash)
    • SCR34: Calculating size and position in a way that scales with text size (Scripting) OR
  • G206: Providing options within the content to switch to a layout that does not require the user to scroll horizontally to read a line of text

New Techniques

  • Using fluid grids (CSS). Ref: fluid grids
  • Using em based media queries (CSS). Ref: em based media queries
  • Using responsive images (CSS). Ref: HTML picture element and srcset attribute

Advisory

None

  • @@ Using fixed sized containers and fixed position content (CSS)

  • @@ (HTML) Using preformatted text or excluding preformatting text as an exception to no two dimensional scrolling.
