Failure of Success Criterion 1.3.5 due to use missing or incorrect autocomplete attributes

Metadata

Success Criterion 1.3.5

Failure

Applicability

This technique applies to form inputs created in markup languages that support any attribute that allows the programmatic association of the defined token values found in Section 7 of the WCAG 2.1 Specification.

Description

The purpose of this technique is to identify a failure condition where form inputs do not have the correct autocomplete attributes for inputs that request information from the user of the form.

When Success Criteria 1.3.5 was created there was only one known technique to apply the appropriate token value to a form input: the autocomplete attribute. It is anticipated that in the near future other techniques will emerge that will afford a similar ability, potentially more specifically designed for the purpose.

Success Criteria 1.3.5 uses the autocomplete attribute’s fixed list of token values because the programmatic association of specified token values (metadata) allows for other machine processing, such as expressing the input label in different modalities.

An important additional requirement is that the token values are associated exclusively to inputs that are scoped directly to the end user.

The autocomplete attribute was first published in the HTML 5.0 specification, which states:

User agents sometimes have features for helping users fill forms in, for example pre-filling the user's address based on earlier user input. The autocomplete content attribute can be used to hint to the user agent how to, or indeed whether to, provide such a feature.

For the autocomplete function to pre-fill inputs with the correct user-data, that user-data must first be stored on the user’s local machine (...based on earlier user input...). This is why autocomplete cannot pre-fill forms accessed via public terminals or other user-agents that have not previously stored the corresponding values.

While some user-agents or helper-applications may store more than one user profile, it is assumed that if only one user-profile exists, the stored data will be directly associated to the actual or primary user. For example, on Robert Smith's device(s) the data stored (name, address, email, phone, etc.) will be that of Robert, and not of Susan Jones or even of Robert’s son Steven.

Examples

Missing attribute

An online form used to collect the user's name and birthday which does not include the autocomplete attributes at all.

<h2>Your details:</h2>
<label for="name1">Name:</label>
<input type="text" name="field1a" id="name1"> 
<label for="birthday1">Birthday:</label>
<input type="text" name="field1b" id="birthday1>

Incorrect attribute

An online form used to collect the user's name and birthday which uses an incorrect autocomplete attributes. The correct attribute values are name and bday for this example.

<h2>Your details:</h2>
<label for="name1">Name:</label>
<input type="text" name="field1a" autocomplete="yourname" id="name1"> 
<label for="birthday1">Birthday:</label>
<input type="text" name="field1b" autocomplete="birthday" id="birthday1>

Autocomplete off

An online form used to collect the user's name and birthday which uses an autocomplete value of "off".

<h2>Your details:</h2>
<label for="name1">Name:</label>
<input type="text" name="field1a" autocomplete="off" id="name1"> 
<label for="birthday1">Birthday:</label>
<input type="text" name="field1b" autocomplete="off" id="birthday1>

Tests

Procedure

For each form field:

  1. Check that the form field collects information related to the user who is filling in the form.
  2. Check that the form field does not has a valid and well-formed autocomplete attribute and value pair that matches the purpose of the input.

Expected Results

  • If checks #1 and #2 are true, then the failure condition applies, and the content fails the Success Criterion.
