Failure of Success Criterion 1.3.5 due to use of autocomplete attribute on inputs not about the primary user

Metadata

Success Criterion 1.3.5

Failure

Applicability

This technique applies to form inputs created in markup languages that support any attribute that allows the programmatic association of the defined token values found in Section 7 of the WCAG 2.1 Specification or future versions of that specification.

Description

The purpose of this technique is to identify and test a failure condition where form inputs are not properly scoped to the actual user of the form.

At the date of creation of Success Criteria 1.3.5, there is only one known technique to apply the appropriate token value to a form input: the @autocomplete attribute. It is anticipated that in the near future other techniques will emerge that will afford a similar ability, potentially minus the specific functionality originally envisioned for the attribute.

Success Criteria 1.3.5 expands on the original functionality of @autocomplete and that attribute’s fixed list of token values, because while the fixed list of tokens are key to the auto-filling function, the programmatic association of specified token values (metadata) to their related inputs also allows for other machine processing, such as expressing the input label in different modalities.

Importantly however is the additional requirement that the token values are associated exclusively to inputs that are scoped directly to the end user, as opposed to any or all form inputs found on a form.

The @autocomplete attribute was first published in the HTML 5.0 specification, which states:

User agents sometimes have features for helping users fill forms in, for example prefilling the user's address based on earlier user input. The autocomplete content attribute can be used to hint to the user agent how to, or indeed whether to, provide such a feature.

However for the @autocomplete function to be able to pre-fill specifically tagged inputs with the correct user-data, that user-data must first be stored on the user’s local machine ("...based on earlier user input..."). This is why @autocomplete still cannot pre-fill forms accessed via public terminals or via user-agents that have not previously stored the corresponding values.

Additionally, while some user-agents or helper-applications may store more than one user profile, it is presumed that if only one user-profile exists, the stored data will be directly associated to the actual or primary user: i.e. on Robert Smith’s device(s) the data stored (name, address, email, phone, etc.) will be that of Robert, and not of Susan Jones or even of Robert’s son Steven.

Given this presumption, Success Criteria 1.3.5 is specifically scoped only to inputs directly related to the primary user, and not to other users, as applying the token terms to inputs not related to the primary user could potentially result in forms being auto-filled with bad data-values.

Tests

Procedure

For each form field:

  1. Check that the form field collects information related to a subject who is not the primary user of the form.
  2. Check that the form field has a valid and well-formed autocomplete attribute and value pair.

Expected Results

  • If checks #1 and #2 are true, then the failure condition applies, and the content fails the Success Criterion.
